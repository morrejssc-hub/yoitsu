# 0004: 预算作为预测模型 (Budget as Prediction)

## 1. 现状与存在的问题
早期版本中，预算是一个硬性运行时约束——一旦成本超出分配值，Job 会被强制终止。这导致两个结构性矛盾：

1. **方差信号被截断**：如果 Job 在成本触及上限时被强杀，系统只能记录"到了上限被杀"，永远无法知道"如果让它跑完，会偏离预测多少"。这使得自优化循环（ADR-0010）丧失了衡量预测精度的数据来源。
2. **惩罚模型与预测目标冲突**："迭代惩罚模型"将额外迭代转化为虚拟成本惩罚，依赖于一个成本上限来产生收敛压力——但成本上限本身已被挂起。

## 2. 做出的决策与原因

### 2a. 预算是预测，不是执行约束
Planner 在 Spawn 子任务时提供的 `estimated_budget` 是一个**预测值**——"我估计这个任务大概要花多少"。运行时**不会**因为成本超出该值而终止 Job。实际花费不设上限，使得 `observation.budget_variance` 能捕获到真实偏差。

### 2b. 分解约束通过结构手段实现
鼓励 Planner 将大任务拆小不是靠运行时掐断执行，而是靠两个 Spawn 时的静态校验：
- `min_cost`：低于此值的 Spawn 被拒绝（任务太碎）。
- `max_cost`：高于此值的 Spawn 被拒绝（任务太大，应拆分）。

这两个都是 **Task 创建时的前置校验**，不是运行时行为。

### 2c. 系统后备线保持活跃
以下后备机制与预算模型无关，始终工作：
- `max_iterations_hard`：绝对迭代上限，防止死循环。
- `job_timeout`：壁钟超时，防止进程挂死。
- `tool_timeout_seconds`：单工具调用超时，防止阻塞。

它们是**操作安全网**，不是分解引导机制。

## 3. 期望达到的结果
- 自优化循环拿到的预算偏差数据是**未截断的真实值**，可用于评估 Planner 的分解与预测质量。
- 预算预测精度成为系统健康度的代理指标（见 ADR-0010）。

## 4. 容易混淆的概念
- **预测 (Prediction) vs 强制 (Enforcement)**
  - `estimated_budget` 是预测值。`max_iterations_hard` 是强制值。前者用于优化反馈，后者用于防止灾难。
- **Spawn 时校验 vs 运行时截断**
  - `min_cost` / `max_cost` 在 Task 创建时检查，不是 Job 执行中检查。

## 5. 挂起决策日志

| 日期 | 影响的决策 | 原因 |
|------|-----------|------|
| 2026-03-31 | 成本强制执行 → 挂起 | 截断了方差数据，损害自优化信号质量 |
| 2026-03-31 | 迭代惩罚模型 → 挂起 | 依赖外部已不存在的成本上限进行收敛 |
| 2026-03-31 | 屏障函数 (Barrier) → 挂起 | 依赖成本上限产生收敛压力 |

> [!NOTE]
> 以上挂起项仅在未来重新引入成本强制执行时才需要重新评估。当前系统不应尝试恢复它们。
