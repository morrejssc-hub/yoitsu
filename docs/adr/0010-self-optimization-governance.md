# 0010: 自优化治理 (Self-Optimization Governance)

## 1. 现状与存在的问题
系统长期目标是自我改进：识别自身运行中的低效并逐步修复。过去的尝试（palimpsest-v1）依赖单 Prompt 的自我反思模式——Agent 审视自己的事件历史并撰写自己的记忆。这已被证明不足：单个 Prompt 无法真正评估自己的行为，且自撰记忆的可靠性有限。

Yoitsu 的架构天然分离了观察者（Trenni）与执行者（Palimpsest），提供了外部评估的结构前提。剩下的问题是：如何将观察信号转化为行动，且不构建特殊的优化子系统。

## 2. 做出的决策与原因

### 2a. 核心治理哲学：优化是普通任务
**没有特殊的"优化模式"，没有特权访问，没有独立调度器。** 优化任务与外部提交的工作任务使用完全相同的管道（Planner → Worker → Evaluator），竞争相同的资源。

**原因**：特殊通道会引入复杂性、测试盲区和特权升级风险。将优化任务"降格"为普通任务反而使其天然享有系统已有的所有隔离、评估和版本控制能力。

### 2b. 闭环：结构化信号 → 累积触发 → Review → Proposal → 执行
```
观察信号 (observation.*) ──累积阈值──→ Review Task ──产出──→ Improvement Proposals
                                                            │
                                                            ↓
                                                   普通优化 Task（修改 evo/）
                                                            │
                                                            ↓
                                                   预算预测精度变化（效果验证）
```

1. **结构化观察信号**：由 Trenni 和工具网关按确定性条件机械发出（`observation.budget_variance`、`observation.tool_retry` 等），不涉及大模型。
2. **累积触发**：Trenni 的 Trigger 规则在 N 条匹配事件累积后自动创建 Review Task（默认 `accumulate: 20`，简单批处理语义）。
3. **Review Task**：读取近期观察事件，识别重复模式，产出具体的改进提案。
4. **优化 Task**：每个提案成为独立 Task，通过正常 Spawn 执行，修改 `evo/` 下的角色、Prompt、工具等。
5. **效果验证**：优化执行后，预算预测精度（`budget_variance`）是否改善作为反馈信号。

### 2c. 预算预测精度作为系统健康代理指标
Planner 的 `estimated_budget` 与实际成本的偏差比率是一个归一化的、跨任务类型可比较的连续信号：
- **系统性低估** → 角色职责不清、Preparation 缺少上下文、分解粒度太粗
- **系统性高估** → Preparation 冗余、角色能力被低估、任务过度拆分
- **特定类型高方差** → 该领域的上下文函数或角色定义有弱点

优化目标不是"降低成本"，而是"提高预测精度"——后者结构上隐含了前者的大部分改进方向。

### 2d. 信号先于行动
观察事件的 Schema 应尽早定义并发出（Phase 1-2），但优化任务应等到信号量充足后才触发（Phase 5）。稀疏信号下的优化会产生虚假模式和过早变更。

**Phase 分期**：
- **Phase 1-2**：定义 `observation.*` Schema，在相关代码路径中发出信号
- **Phase 3**：Reviewer 角色可读取观察事件作为额外上下文
- **Phase 4**：Pasloe 查询能力支持时间窗口聚合
- **Phase 5**：Trigger 规则激活 Review Task，闭环建立

## 3. 期望达到的结果
- 系统自身的改进能力随运行时间自然增长，不需要人工介入识别问题。
- 所有优化变更都经过正常的 Task 评估流程，具备可审计性和回滚能力。

## 4. 容易混淆的概念
- **观察信号 vs AI 生成总结**
  - 观察信号是确定性的机械产出（tool retry 了 3 次、预算偏差了 40%），不是大模型写的"我觉得这次做得不好"。
- **Review Task vs 一般 Agent 自我反思**
  - Review Task 是一个独立的 Task，由外部触发，读取他者（其他 Job）的数据来判断。不是同一个 Agent 反思自己的上一轮。
