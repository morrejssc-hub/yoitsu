# 0006: 任务级自动发布 (Task-Level Publication)

## 1. 现状与存在的问题
当前体系下的“发布 (Publication)”操作停留在单次 Job（工作单元）的基础层面上——Worker 一旦运行结束，底层机制就会机械地自动将产出提交并推送到某个 Git 分支上。
这导致系统的闭环极其割裂：被创建出的分支孤立存在，系统无法自动在“任务整体完成”语义层面上发起代码拉取请求（PR）提交。同时，若企图将 PR 逻辑硬塞入底层的 `publication_fn` 钩子或调度器 (Supervisor) 中，会极度加深调度侧对 Git 平台的耦合度，且无法在包含子任务的复杂分发网络中找到恰当的触发点。

## 2. 做出的决策与原因
**决策**：将具体的“合并交付物 (例如创建 PR)”这一行为从执行器的底层系统级钩子中剥离出来，封装为一个具备外部交互能力的模型工具（名为 `create_pr`）。该工具仅向在父任务汇总阶段（Join Mode）运行的 `Planner` 角色开放授权调用。
**原因**：合并与交付本质上是智能体主导的“主观战术动作”，而非底层的纯硬件状态固化。Planner 在汇总阶段已经握有全部子任务验证器 (Evaluator) 的执行裁判结果与所在分支位置，由它来决定“是否合并”、“一次要拉几条分支”、“写什么样的 PR 报告”是在业务逻辑层面上最理直气壮的架构做法。

## 3. 期望达到的结果
- 调度器和执行器框架不再硬绑定各类代码托管平台的“如何提合并”API，彻底保持环境底层不可知的整洁性。
- 系统完成由单体变更推送 (Branch Push) 到语义功能闭合交付 (PR Create) 的自动化跃升。
- 当 PR 本身遭遇权限或网络报错时，不会错误地将底层无状态的 Job 标为失败状态，保障作业的忠实结案。

## 4. 容易混淆的概念
- **作业级发布 (Job-Level Publication) vs 任务级发布 (Task-Level Publication)**
  - **作业级发布** (`publication_fn`) 只是机械式的归档行为，负责执行结束后不问对错地固定数据或推送分支。
  - **任务级发布** (通过 `create_pr` 工具) 是 Planner 通过审视上述归档记录，深思熟虑后主动发起的向上反馈过程。两者毫不冲突且同时存在：Worker 创造归档 (分支)，Planner 消费归档 (拉 PR)。
