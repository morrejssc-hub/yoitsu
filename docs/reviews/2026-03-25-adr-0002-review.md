# ADR 0002 最新实现评审

> 日期: 2026-03-25
> ADR: `docs/adr/0002-task-orchestration-decoupling.md`
> 范围: `yoitsu`, `trenni`, `palimpsest`, `yoitsu-contracts`
> 关注点: 功能设计完整性、任务/作业解耦、代码简洁性与边界收口

## 评审结论

这轮改造已经把核心方向切对了: `trenni` 开始接管 Task 生命周期，`palimpsest` 不再直接发 `task.updated`，共享 contracts 也已经换到 `trigger.*` 和 `task.*` 生命周期事件。

但当前状态仍然是一次**半迁移**:

- 内核层已经按 ADR 0002 改了
- 外围入口、观测面、join 上下文、异常恢复路径还停留在旧语义

这导致系统在“正常 happy path”之外，会出现明显的功能断层。下面按严重程度给出结论。

## 关键问题

### P0: 顶层提交入口仍然发旧的 `task.submit`，而不是新的 `trigger.*`

**现象**

- `yoitsu submit` 仍然发送 `task.submit`
- `scripts/submit-tasks.py` 也仍然发送 `task.submit`
- payload 仍然是旧的扁平结构，核心字段叫 `task`，而不是 `goal`

**为什么这违反 ADR 0002**

ADR 0002 已经把外部入口改成 `TriggerData` (`trigger.*`)，并明确废弃 `TaskSubmitData`。但现在最上层用户入口没有迁移，意味着新架构没有真正贯穿到系统边界。

**影响**

- 从根仓库 CLI 提交任务时，`trenni` 当前实现根本不会消费这些事件
- 就算把事件类型手工改成 `trigger.external`，现有 payload 结构也不兼容 `TriggerData(goal, context)`
- 这不是“文档没更新”，而是用户主路径实际失效

**证据**

- `yoitsu/cli.py` 仍在发 `task.submit`
- `scripts/submit-tasks.py` 仍在发 `task.submit`
- `yoitsu-contracts` 已把入口 contract 改成 `TriggerData(goal, context)`
- `trenni` 只处理 `trigger.*`

**建议**

不要在多个调用方各自“手拼”新 payload。应把“任务 YAML/CLI 输入 -> TriggerData”收敛成一个单独的 adapter/translator，作为唯一入口边界。

### P1: `runtime_lost` / replay 恢复路径只补 `job.failed`，不会把 `task` 收口成终态

**现象**

`trenni` 现在只有在 `handle is not None` 时才调用 `_evaluate_task_termination()`。但在两条关键恢复路径里，handle 已经不存在了:

- checkpoint reaper 先把 job 从 `self.jobs` 移除，再补发 `job.failed`
- replay 发现容器消失时，直接补发 `job.failed`

后续这个 `job.failed` 再被消费时，`handle` 仍然是空，因此 task 不会转成 `task.failed`。

**为什么这违反 ADR 0002**

ADR 0002 的核心不是“多几个事件名”，而是 **Task 的结构化终止完全由 Trenni 判断**。现在一旦进入恢复/补偿路径，Task 终态判断就丢了，直接破坏了结构化终止和可重放性。

**影响**

- task 可能永久停留在 `submitted`
- 依赖该 task 的条件判断可能永远不满足
- restart 后的恢复结果不再可预测，违背 ADR 0002 的 determinism 目标

**证据**

- `_handle_job_done()` 只在 `handle is not None` 时评估 task 终态
- `_checkpoint()` 的 `runtime_lost` 分支只补 `job.failed`
- `replay.py` 的容器丢失分支同样只补 `job.failed`
- 现有测试只验证发出了 `job.failed(code=runtime_lost)`，没有验证 `task.failed`

**建议**

把“job 终态 -> task 结构化收口”的逻辑抽成独立函数，和 runtime handle 是否还在彻底解耦。只要 job 进入 terminal，task 是否 terminal 就应该始终被重新计算。

### P1: Join 上下文仍然读取已废弃的 `task.updated`，导致 join job 失去子任务结果

**现象**

`palimpsest/evo/contexts/loaders.py` 里的 `join_context()` 仍然只查询 `task.updated`，并从中提取 `status` 和 `summary`。但 ADR 0002 后这类事件已经不存在了。

**为什么这违反 ADR 0002**

ADR 0002 不是只把“谁发 task 状态”改掉，而是把 Task 观测模型改成显式生命周期事件。join 上下文作为父任务继续决策的输入，却仍绑在旧事件上，等于把旧架构偷偷留在了运行时上下文层。

**影响**

- join job 会按条件被调度起来
- 但它读不到任何真实的 child task 状态和 summary
- 结果就是 parent continuation 失去最关键的输入，只能“盲续”

**证据**

- 默认 role 明确启用了 `join_context`
- `join_context()` 仍然 fetch `task.updated`
- `spawn_handler` 已经把 join job 正式建模并注入 child task ids

**建议**

这里不要继续让 `palimpsest` 自己拼 task 语义。更优雅的做法有两种:

1. `join_context` 改为消费新的 `task.created/completed/failed/cancelled` + child job terminal events
2. 更进一步，由 `trenni` 在 join job 的 `job_context` 里直接提供归一化后的 child results，runtime 只负责展示，不负责推理

第二种更符合 ADR 0002 的权责分离。

### P1: 入口文档和观测脚本仍然描述旧架构，削弱了 ADR 0002 的“可观测性提升”

**现象**

- 根 README 仍然宣称 Task 有 `in_progress / blocked / needs_review / complete`
- `docs/architecture.md` 仍写着 runtime 发 `task.updated`
- `palimpsest/docs/design.md` 也仍保留旧模型
- `scripts/monitor.py` 仍然只把 `task.submit` 当成任务入口统计

**为什么这重要**

ADR 0002 里把 observability 单独写成了一个核心问题。现在代码已经切到新事件模型，但运维入口和架构说明还在讲旧故事，团队会继续围绕错误的系统边界思考和排障。

**影响**

- 新人理解的系统边界是错的
- 监控会漏掉新的触发事件
- review / debug 时很难区分“实现 bug”和“认知模型过期”

**建议**

把 README、architecture、monitor 三处视为一个统一收口面，一次性改到位，不要再允许旧术语继续外溢。

## 设计与优雅性问题

### P2: `palimpsest` 的 publication 阶段还残留旧的 `task_status` 语义分支

`publish_results()` 仍然读取 `result["task_status"]` / `result["status"]`，并根据 `in_progress` / `blocked` / `needs_review` 选择 commit prefix。现在 interaction loop 已经不再返回这些字段，所以这里成了“看似兼容、实则误导”的残留层。

这不是当前最致命的问题，但它会把已经删掉的 task 语义继续偷偷留在 runtime 里，和 ADR 0002 的方向相反。

**建议**

publication 只关心 job 结果和 summary，不再感知 task 语义。

### P2: `trenni/supervisor.py` 有重复的 `match case` 分支，说明收口还不够干净

`_handle_event()` 中同一组 `job.spawn.request` / `job.completed` / `job.started` / `supervisor.job.launched` 分支重复出现了一次。虽然不一定造成功能错误，但这是明显的 merge residue。

这种问题本身不严重，但它暴露出这次重构还没有经过一次“按新模型整理代码”的最后清洗。

## 总体判断

这次 ADR 0002 改造的**方向是对的**，而且核心思想也已经进入代码主干了；问题主要集中在“系统边界没有一起迁移”。

换句话说，现在不是架构想法出了问题，而是迁移收口还差最后一轮:

- 入口层没有切完
- 恢复路径没有切完
- join 语义没有切完
- 文档和观测面没有切完

如果先把这四块补齐，整个系统的解耦程度和优雅性会明显上一个台阶。

## 建议的收口顺序

1. 先修提交入口，把 `yoitsu submit` / `scripts/submit-tasks.py` 全部切到 `TriggerData`
2. 再修 `runtime_lost` / replay 的 task 终态收口
3. 然后重做 join context，让父任务继续执行时真正拿到 child 结果
4. 最后统一清理 README / architecture / design / monitor / 残留旧字段
