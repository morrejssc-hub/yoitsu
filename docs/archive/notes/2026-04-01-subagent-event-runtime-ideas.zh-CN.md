# 子代理笔记：事件溯源的单任务运行时构想

日期：2026-04-01
来源：一次不带当前实现上下文的子代理架构探索
目标：挑战现有“四阶段”框架，并从第一性原理提出替代方案

## A. 根本设计原则

1. 权威状态只能放在可重放的事件和可引用的产物里，不能放在进程内返回值、对象图或工作区目录里。
2. 运行时内核应该只做两件事：根据已知事实形成当前视图；基于当前视图选择下一次可执行动作。
3. 任何会影响后续决策的非确定性观察，都必须被事件化，或通过 artifact 引用被冻结下来；否则重启后无法恢复出等价行为。
4. 本地工作区属于“物化层”，不是“真实源”。它可以加速执行、承接副作用，但默认应被视为可丢弃、可重建。
5. 外部副作用必须穿过显式的 effect 边界，并带上因果 ID、幂等键、前置条件和结果回执，以便重试和审计。
6. prompt、context gathering、tooling、publication 都应是可替换的策略或能力，不应被硬编码进运行时骨架。
7. 错误处理、恢复、重试、人工介入、超时，不应是旁路控制流，而应是同一套任务事件模型中的正式事件。
8. 系统应明确区分“权威状态”和“加速状态”。
权威状态：event store 加 artifact 引用。
加速状态：workspace、cache、runtime memory、materialized views。

## B. 三种真正不同的高层架构模型

1. 事件驱动决策循环
2. 派生图 / 能力图
3. 黑板式义务-证据-行动

## C. 各模型展开

## 模型 1：事件驱动决策循环

核心抽象单位：一次“决策步”或一次“effect attempt”。

状态面：
- `event store` 保存任务日志和唯一权威的因果链。
- `artifact store` 保存不可变的上下文包、外部快照、工具输出和发布回执。
- `local workspace` 只在某些 effect 需要文件系统时被租用和物化。
- `runtime memory` 只保存当前投影视图、句柄和缓存。

流转：
- 从事件流投影出 `TaskView`。
- 由策略选择下一步 `EffectSpec`。
- 执行该 effect。
- 将观察、artifact 引用、结果摘要、错误和重试决策写回事件流。
- 重新投影并继续。

必须事件化的内容：
- 任务接收
- 输入引用
- 关键观察
- 每次 effect 的开始和结束
- 失败原因
- artifact 引用
- 最终完成或放弃
- 人工介入

可以只留在本地的内容：
- 可重建缓存
- 临时 clone 目录
- token 流
- 纯加速性质的中间文件
- 不会影响后续决策的 scratch 输出

错误恢复与重试：
- 恢复边界是 effect，而不是整个任务。
- 每次 effect 都有幂等键。
- 重启后先重放事件，再判断哪些 effect 已完成，哪些可以安全重试，哪些需要补偿。
- 恢复的本质是“从日志继续选下一步”，而不是“恢复进程内状态”。

对非 git-native / 外部系统任务的适配性：
- 很强。
- git 只是 effect 的一种类别，和调用 API、驱动浏览器、发工单、抓取外部数据没有本质区别。

优点：
- 内核最小
- 恢复模型清晰
- 事件语义强
- 对外部任务通用性好
- 容易把可观测性、审计、重试做成一等能力

缺点：
- 如果任务内部包含大量可并行、可复用的确定性子计算，纯循环模型的表达力不如图模型。
- 对 projection 和 effect schema 的设计要求较高。

## 模型 2：派生图 / 能力图

核心抽象单位：一个可执行节点，表示“通过某个 capability invocation，从这些输入派生这些输出”。

状态面：
- `event store` 记录图结构变化、节点调度、节点尝试、失效与完成。
- `artifact store` 保存节点输入输出、快照和上下文包。
- `local workspace` 是某个节点的局部执行沙箱。
- `runtime memory` 只维护 ready set、依赖计数和调度缓存。

流转：
- 任务从目标和种子输入开始。
- 运行时逐步构造或扩展图。
- 当某个节点依赖满足时执行它。
- 节点输出变成 artifact，并解锁下游节点。
- 失败节点可以单独重试、失效或替换。
- 图本身是持久对象。

必须事件化的内容：
- 节点定义
- 依赖关系
- 节点版本
- 输入输出引用
- 失效关系
- 接受 / 拒绝决定

可以只留在本地的内容：
- 节点内部临时文件
- 可重复下载缓存
- 局部推理 scratch 数据

错误恢复与重试：
- 以节点为单位重试。
- 失败不会污染整张图。
- 如果上游观察发生变化，可以把相关子图标记为 invalidated，并局部重算。
- 重启时通过事件重建图和 ready set。

对非 git-native / 外部系统任务的适配性：
- 适合，尤其适用于“上下文组装 + 多种检查 + 多种发布”的复合任务。
- 外部系统调用也可以建模为节点，但需要认真处理副作用节点的幂等性。

优点：
- 很强的 lineage 能力
- 支持局部重算
- 支持结果复用
- 支持并行扩展
- 对确定性子结构很友好

缺点：
- 对开放式 agent loop 不够自然
- 容易把探索性工作硬塞进 DAG
- 当目标频繁漂移时，图会快速膨胀并带来管理复杂度

## 模型 3：黑板式义务 / 证据 / 行动

核心抽象单位：黑板上的一项条目，例如义务、问题、假设、证据记录、提案或异议。

状态面：
- `event store` 本身就是黑板日志。
- `artifact store` 保存原始证据、上下文包、执行产物和外部回执。
- `local workspace` 供某个 specialist 处理特定义务时使用。
- `runtime memory` 只保存当前优先级、选择启发式和局部工作集。

流转：
- 任务启动时注入初始义务，例如确认目标、收集约束、产生候选动作、验证外部可达性。
- 每轮挑选一个尚未解决的义务。
- 某个 specialist 产出证据或提案。
- 提案可以被接受、驳回，或进一步细化为新的义务。
- 当没有关键未决事项时流程结束。

必须事件化的内容：
- 义务创建
- 优先级变化
- 证据引用
- 提案
- 接受 / 驳回
- 冲突
- 人工裁决

可以只留在本地的内容：
- specialist 的 scratch 推理
- 临时缓存
- 非关键临时文件

错误恢复与重试：
- 失败不会只是简单变成“step failed”，而是形成新的未决义务或异议。
- 重试表现为：重新为该义务派生证据，或提出替代提案。
- 重启时只需从事件中重建未解决义务集合。

对非 git-native / 外部系统任务的适配性：
- 很强，尤其适合目标模糊、约束多变、需要人机协作的真实外部任务。

优点：
- 最不受四阶段思维束缚
- 能自然表达不确定性、争议和等待人工判断
- 适合复杂现实任务

缺点：
- 实现和观测门槛最高
- 对简单任务显得过重
- 吞吐与直觉性不如前两种模型

## D. 是否应该坚持“四阶段”作为主抽象？

不应该。

“四阶段”最多适合作为某类任务的视图、策略模板或默认 profile，不应该成为运行时内核抽象。

原因：
1. 它把时间顺序误当成了本体结构。
2. 它默认任务总能被稳定切成几个边界清晰的大块，而真实外部任务经常做不到。
3. 它弱化了回退、反复观察、局部重试和人工介入这些真实控制流。
4. 它很容易把编辑器 / CI / git 任务的惯性带到所有任务上。

建议替代方案：
- 将主抽象替换为“事件化的下一动作选择”。
- 内核采用决策循环。
- 再在其上叠加图模型或黑板模型作为更高层的任务表达。

## E. 最推荐的模型

最推荐：事件驱动决策循环。

原因：
- 它是最小、最稳的内核。
- 它最符合目标状态：
  - 单任务 agent loop
  - 事件系统作为真实源
  - 工作区是非权威的物化层
  - git 只是可选项，不是前提
  - 恢复与可观测性是一等公民

一个简洁但具体的流程示意：

1. `TaskAccepted`
- 记录任务目标、输入引用、策略配置和初始约束。

2. `ProjectTaskView`
- 从事件流构建当前视图。
- 导出已知事实、未决问题、可用 artifact 和重试预算。

3. `SelectNextEffect`
- 由策略选择下一步，例如：
  - `AcquireObservation`
  - `AssembleContextBundle`
  - `RunAgentTurn`
  - `InvokeExternalTool`
  - `ValidateOutcome`
  - `PublishResult`

4. `ExecuteEffect`
- 如果需要文件系统，就租用一个 workspace 并物化必要输入。
- 大对象写入 artifact store。
- 摘要和引用写回事件流。

5. `RecordOutcome`
- 成功时，记录 observation、artifact 或 receipt。
- 失败时，记录错误类别、是否可重试、建议回退点和补偿数据。

6. `Reproject`
- 从事件流重新计算任务视图。
- 如果验证失败，就回到新的观察或新的 agent turn，而不是硬跳到某个固定阶段。

7. `Terminate`
- 当出现 `CompletionAccepted` 或 `AbandonedWithReason` 时结束。
- 最终输出是事件链和 artifact 引用，而不是进程返回值，也不是工作区目录。

解释：
- preparation、interaction、publication 仍然可以作为某种视图存在。
- 但它们不应该定义内核本身。

## F. 最容易被编辑器 / CI / Git 思维误导的地方

1. 把工作区当成真实源。
实际上它只是当前 effect 的物化环境；真正需要跨重启保留的是事件日志和 artifact 引用。

2. 把 git commit、PR、diff 当作默认输出。
真实外部任务的输出可能是 API 回执、表单更新、报告、消息、截图、数据库记录或人工确认。

3. 把“准备-执行-发布”当作普适本体。
许多任务实际是：观察、局部行动、验证、再观察，而不是单向流水线。

4. 把日志当作旁路，而不是把执行事实本身事件化。
如果可观测性与恢复不共用同一套事件模型，故障时两边都会不可信。

5. 把函数返回值或内存对象当成恢复点。
重启后真正可靠的只有事件日志、artifact 和外部回执。

6. 把外部读取当作可以随时重跑的纯函数。
很多外部系统是时变的；只要某次读取影响了后续决策，就应当做快照或至少做版本化。

7. 把重试理解为“整任务再跑一遍”。
更合理的是 effect 级、节点级或义务级重开，并保留完整因果链。

8. 把 agent transcript 当作状态本体。
真正需要持久化的是观察、提案、artifact 引用和工具回执，而不是无限增长的对话文本本身。
